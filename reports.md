# mse1h2026-helper

## О проекте

### Постановка задачи

#### Проблема, которую решает проект
При приёме лабораторных и курсовых работ довольно много времени уходит на анализ кода студентов. Также время преподавателя может быть ограничено, если его подключают в качестве "пожарного" человека на комиссии или пересдаче. Существует много типовых замечаний, которые проскальзывают в каждой второй работе. Есть редкие вещи, которые один раз находятся - и их хорошо было бы записать. Требуется система, которая принимает на вход студенческую работу, а на выходе выдаёт список замечаний/вопросов, которые студенту можно задать.

#### Краткое описание задания
Создать инструмент, который получает на вход pull-request'ы с кодом на Си/Питоне, а на выходе генерирует отчёт, в котором указаны проблемы в данном коде.

#### Результат
Консольное приложение, обернутое в Docker, работающее в неинтерактивном режиме (на вход - данные для авторизации, настройки, список PR; на выходе - отчеты по каждому PR).

### Требования
1. Язык: Python.
2. Поставка: Docker.
3. Типа UI: CLI (консольное приложение).
4. Запуск: docker run program [args...].
5. Режим работы: неинтерактивный, на конечной машине пользователя, без удалённого сервера, без работающего на фоне сервиса.
6. Функциональность:
	- Авторизация на GitHub и Forgejo.
	- Загрузка PR.
	- Получение списка проблем от линтеров при анализе загруженных файлов.
	- Анализ проблем линтеров и преобразование их в вопросы.
	- Получение списка вопросов от собственных правил при анализе загруженных файлов.
	- Структурирование списка вопросов из всех источников.
	- Вывод списка вопросов конечному пользователю.
7. Поддержка линтеров: Pylint для Python и OCLint для C.
8. Собственные правила анализа:
	- Каждое правило реализуется как функция или класс, принимает представление кода (AST, список файлов, список коммитов) и возвращает вопросы.
	- Возможность легко добавлять новые правила на уровне кода (например, с помощью модуля utils с реализацией типовых операций).
9. Настройки:
	- Уровень строгости анализа: error, warning, note.
	- Настройки линтеров.
10. Выходные данные:
	- Plain text с вопросами, разделёнными по источникам (линтер/собственные правила), с указанием файлов, номеров строк, проблемных конструкций и, возможно, типовых вопросов, указанных для созданных правил (например, "Почему вы используете эту конструкцию?").
11. Документация:
	- Инструкция по установке и запуску приложения.
	- Примеры написания и подключения новых правил.
	- Список поддерживаемых библиотек C для анализа программ, которые их используют.

### Сценарии использования
**Категории пользователей:**
- Преподаватель: использует инструмент для автоматической проверки работ студентов, что включает разработку собственных правил, запуск инструмента с заранее разработанными или собственными правилами и получение отчёта, содержащего список вопросов для задания студентам. Ему важно, чтобы у программы были простой интерфейс и понятный вывод, чтобы тратить как можно меньше времени и сил на использование инструмента.
- Студент: является опосредованным пользователем, поскольку на него влияют результаты работы программы. Ему важно, чтобы вопросы были справедливыми и чтобы количество ложноположительных обнаружений было сведено к минимуму.
- Деканат: может быть опосредованным пользователем в случае, если программа в результате работы выведет преподавателю некорректные вопросы, поскольку им придётся разбираться с такой ситуацией. Важно, чтобы можно было регулировать сложность задаваемых вопросов и чтобы сложность вопросов выводилась в отчёте.

### Сценарий 1 - Создание собственного правила

**Действующее лицо:** преподаватель.

**Основной сценарий:**
1. Преподаватель создаёт файл с новым правилом анализа.
2. Преподаватель использует функции модуля `utils` для реализации нового правила.
3. Преподаватель указывает созданный файл в аргументах запускаемой программы с помощью механизма bind-mount в Docker.

**Альтернативный сценарий:**
1. Преподаватель после создания файла правила добавляет его в репозиторий и пересоздаёт образ Docker.
2. От преподавателя больше не требуется указывать созданный файл в аргументах запускаемой программы.

**Результат:** правило создано и добавлено в качестве аргумента к запуску программы.

### Сценарий 2 - Запуск программы для анализа пулл-реквеста

**Действующее лицо:** преподаватель, студент.

**Предусловие:** преподаватель создал токен на GitHub, который позволяет авторизоваться и загрузить пулл-реквесты студентов.

**Основной сценарий:**
1. Преподаватель запускает контейнер с программой, указывая токен, репозиторий, номер пулл-реквеста и другие аргументы.
2. Программа авторизуется в GitHub с помощью токена.
3. Программа загружает ветку, из которой сделан пулл-реквест.
4. Программа запускает линтеры и сохраняет их вывод в своей памяти.
5. Программа анализирует найденных проблем и формирует из них вопросы.
6. Программа запускает собственные правила и сохраняет их вывод в своей памяти.
7. Программа объединяет данные, полученные из линтеров и собственных правил, и структурирует их в форме отчёта.
8. Программа выводит отчёт в поток стандартного вывода.
9. Преподаватель задаёт приведённые в отчёте вопросы студенту.
10. Студент отвечает на заданные преподавателем вопросы.

**Альтернативный сценарий - Ошибка при анализе:**
1. Программа сталкивается с ошибкой (например, нет доступа к репозиторию, отсутствуют поддерживаемые файлы, программная ошибка в линтерах или в коде собственных правил).
2. Программа отлавливает ошибку.
3. Программа формирует сообщение об ошибке.
4. Программа выводит сообщение об ошибке.
5. Программа останавливает свою работу.
6. Преподаватель анализирует ошибку и пытается её исправить.

**Результат:** преподаватель проанализировал пулл-реквест студента и получил список вопросов, которые может использовать для проверки студента.

### Инструкция по запуску
#### Использование готового Docker-образа

1. Скачайте образ из Docker Hub:
```sh
docker pull pasabanov/mse1h2026-helper:latest
```

2. Запустите контейнер, передав ссылку на Pull Request:
```sh
docker run pasabanov/mse1h2026-helper PULL_REQUEST_URL
```

#### Сборка Docker-образа локально

1. Клонируйте репозиторий и перейдите в директорию проекта:
```sh
git clone https://github.com/moevm/mse1h2026-helper
cd mse1h2026-helper
```

2. Соберите Docker-образ из Dockerfile:
```sh
docker build -t mse1h2026-helper .
```

После сборки запустите контейнер, передав ссылку на Pull Request:
```sh
docker run mse1h2026-helper PULL_REQUEST_URL
```
